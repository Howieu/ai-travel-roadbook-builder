# AI Travel Roadbook Builder PRD 与设计文档

日期：2026-07-01

## 1. 产品概述

AI Travel Roadbook Builder 是一个轻量的 Agent 工作流，用于把用户粘贴的旅行攻略、酒店预订信息和交通信息整理成可执行的网页路书。

它不试图替代 Notion、飞书文档、小红书、Google Maps 或订票平台，而是解决旅行规划之后的执行问题：用户已经有很多资料，但出行当天需要的是清晰的时间线、下一站、地图跳转、时间缓冲和备选方案。

## 2. 目标用户

V0 的核心用户是：

经常自由行的年轻用户，尤其是留学生。他们会从小红书、网页攻略、Notion、飞书、Google Maps、朋友推荐和订单确认信息中收集旅行资料，并愿意把文本复制到 AI 助手中，生成一份手机上可打开的旅行路书。

## 3. 问题定义

用户通常不是缺少旅行信息，而是信息太分散：

- 景点、餐厅、购物和避坑信息散落在攻略帖、笔记、聊天记录和地图收藏里。
- 酒店、机票、高铁、火车和巴士信息在不同订单或 App 中。
- Notion / 飞书适合行前规划，但出行当天在长文档里查找信息成本高。
- 用户在路上真正需要快速回答：下一站去哪、几点离开、怎么导航、如果来不及应该跳过什么。

## 4. 产品目标

V0 要验证一个完整闭环：

1. 用户粘贴旅行资料和约束信息。
2. Agent 提取结构化行程数据。
3. Agent 生成 `roadbook.json`。
4. 渲染器生成可分享的静态网页路书。
5. 用户可以在手机上打开路书，按 Day-by-Day 路线执行。

V0 成功的标准不是路线绝对最优，而是能从真实粘贴资料生成一份足够清晰、可演示、可手动微调的网页路书。

## 5. V0 非目标

V0 不做以下功能：

- 自动抓取小红书全文或依赖登录态读取内容。
- 自动导入邮箱、携程、飞猪、航空公司、铁路或酒店账号订单。
- 实时公共交通规划。
- 用户账号系统。
- 多人协作。
- 社区路线分享。
- 支付或商业化。
- 保证生成绝对最优路线。
- 带后端 AI 服务的完整独立 Web App。

这些能力可以在 V0 跑通后再评估。

## 6. V0 输入

V0 采用用户手动粘贴文本的方式，保证生成链路稳定，不依赖不稳定的爬虫、登录或第三方平台接口。

### 6.1 行程基础信息

用户提供：

- 目的地城市或多个城市。
- 开始日期和结束日期。
- 行程节奏：轻松、标准、紧凑。
- 兴趣标签：美食、购物、博物馆、拍照、自然风景、城市漫步、亲子、夜生活等。
- 必去地点。
- 不想去或需要避开的地点。

### 6.2 攻略资料

用户可以粘贴来自以下来源的内容：

- 小红书攻略正文。
- 网页旅行攻略。
- Notion 或飞书笔记。
- 聊天记录。
- 自己整理的备忘录。

链接可以作为来源备注保留，但 V0 不依赖自动读取链接内容。如果链接无法访问，Agent 应提示用户粘贴正文。

### 6.3 酒店信息

每个酒店或住宿点包含：

- 酒店名。
- 地址。
- 入住日期。
- 退房日期。
- 可选备注，例如可否寄存行李、入住时间、最近车站等。

### 6.4 交通信息

每段航班、高铁、火车、长途巴士、渡轮或跨城交通包含：

- 交通类型。
- 出发城市或站点。
- 到达城市或站点。
- 出发时间。
- 到达时间。
- 可选备注，例如航站楼、站台、行李截止时间、提前到达要求等。

## 7. V0 输出

### 7.1 结构化数据

Agent 生成 `roadbook.json`，包含：

- 行程元数据。
- 用户偏好。
- 酒店信息。
- 交通约束。
- 每日路线。
- 每个 stop 的信息。
- 地图搜索词。
- 来源备注。
- 置信度。
- 风险提醒和备选方案。

### 7.2 网页路书

渲染器把 `roadbook.json` 转换为静态 HTML 页面或小型静态站点，复用现有 Travel Roadbook 的视觉语言。

网页包含：

- 行程总览。
- 每日路线卡片。
- 时间线式 stop。
- 酒店和交通摘要。
- 地图跳转按钮。
- 时间缓冲。
- 可跳过或备选建议。
- 必要时展示来源和置信度说明。

页面应优先适配手机，并可作为静态文件或 GitHub Pages 页面分享。

## 8. 用户流程

1. 用户启动 Travel Roadbook Builder 工作流。
2. 如果目的地、日期、节奏或兴趣缺失，Agent 先追问。
3. 用户粘贴攻略资料。
4. 用户粘贴酒店和交通信息。
5. Agent 提取候选地点、餐厅、交通节点、开放时间、预约提醒和风险信息。
6. Agent 只追问会阻塞生成的关键信息，例如旅行日期、到达时间、返程时间。
7. Agent 生成 `roadbook.json`。
8. 渲染器生成 HTML 路书。
9. 用户打开网页检查行程。
10. 用户可继续要求调整，例如“第二天轻松一点”“优先博物馆”“删掉某个景点”。

## 9. 核心需求

### 9.1 信息提取

Agent 需要从粘贴内容中提取：

- 景点。
- 餐厅和咖啡店。
- 商圈和购物点。
- 博物馆和展览。
- 交通节点。
- 开放时间。
- 预约或购票提醒。
- 避坑和风险提示。
- 建议游玩时长。

如果可以定位来源，应保留来源备注，例如“来自用户粘贴攻略第 2 段”。

### 9.2 约束识别

行程必须尽量遵守：

- 到达日和返程日。
- 酒店入住和退房日期。
- 跨城交通。
- 机场、车站和酒店相关时间缓冲。
- 用户选择的节奏。
- 必去地点。
- 避开地点。

默认规则：到达日和返程日不要排得太满。

### 9.3 行程编排

Agent 生成 Day-by-Day 行程，每一天包含：

- 当日标题。
- 必要时分为上午、下午、晚上。
- stop 时间。
- stop 名称。
- stop 类型。
- 简短说明。
- 建议停留时长。
- 关键节点的最晚离开时间。
- 紧凑路线的跳过建议。

V0 不做严格地理最优路径规划。地理排序可以先基于地点名称、城市区域和常识做轻量推理。

### 9.4 地图链接生成

每个 stop 生成地图搜索跳转：

- 高德地图。
- Google Maps。
- Apple Maps。

V0 使用搜索链接，不依赖精确经纬度。

默认优先级：

- 中国大陆行程优先高德。
- 海外行程优先 Google Maps。
- Apple Maps 始终作为 iPhone 用户备用入口。

### 9.5 网页渲染

渲染器需要把 `roadbook.json` 转成 HTML，包含：

- 路线总览页头。
- Day 快速导航。
- 时间线 stop 卡片。
- 地图按钮。
- 酒店和交通摘要。
- 移动端响应式布局。

V0 尽量复用现有 `assets/roadbook.css`。除非静态渲染明显不够用，否则不引入复杂前端框架。

### 9.6 调整循环

生成后，用户可以发起实用型调整：

- 让路线更轻松。
- 增加购物时间。
- 删除某个地点。
- 把必去地点提前。
- 增加餐厅或咖啡店。
- 降低返程日交通风险。

调整时应先更新 `roadbook.json`，再重新渲染 HTML。

## 11. 架构设计

V0 架构：

```text
用户粘贴资料
  -> Agent / Skill 工作流
  -> roadbook.json
  -> 静态渲染器
  -> HTML + 共享 CSS
  -> 本地预览或 GitHub Pages
```

### 11.1 Skill 工作流

Skill 应指导 Agent 完成：

1. 信息 intake。
2. 来源内容解析。
3. 约束信息提取。
4. 缺失信息检查。
5. 行程生成。
6. JSON 校验。
7. 静态网页渲染。
8. 调整循环。

### 11.2 渲染器

渲染器是一个小脚本，负责：

- 读取 `roadbook.json`。
- 校验必填字段。
- 对用户输入文本做 HTML 转义。
- 为每份路书生成 HTML。
- 复用现有 roadbook CSS。

V0 使用 Python 即可，不需要引入前端框架。

## 12. 错误处理

Builder 不应假装不确定的信息已经被验证。

规则：

- 如果旅行日期缺失，必须先追问用户。
- 如果酒店或到达/返程时间缺失，可以生成，但要添加提醒，并避免紧凑排程。
- 如果开放时间未验证，标注“出发前需确认”。
- 如果资料之间冲突，要展示冲突，而不是静默选择。
- 如果攻略链接无法读取，要求用户粘贴正文。
- 如果地点城市或地址不明确，生成地图搜索词并把置信度标为 low。

## 13. 验收标准

V0 完成后至少满足：

- 一段粘贴攻略 + 酒店/交通信息可以生成有效 `roadbook.json`。
- 渲染器可以生成可用 HTML 页面。
- 页面包含行程总览、Day 分组、时间线 stop、酒店、交通和地图链接。
- 每个 stop 至少有一个地图搜索入口。
- 到达日和返程日包含时间缓冲提醒。
- 用户可以要求调整并重新生成路书。
- 生成页面无需登录或后端服务即可查看。

## 14. 后续指标

V0 不要求上线生产级分析，但产品设计应预留以下指标：

- 页面访问量。
- 路线打开次数。
- 地图链接点击量。
- stop 完成点击。
- 用户调整请求类型。
- 生成后满意度。

这些指标用于判断用户是否真的把它当作出行执行工具，而不只是规划文档。

## 15. 路线图

### V0：Agent 工作流 + 静态网页输出

- 粘贴文本 intake。
- 酒店和交通约束解析。
- `roadbook.json`。
- 静态 HTML 渲染。
- 高德 / Google Maps / Apple Maps 搜索链接。
- 一条完整 demo 路线。

### V1：更好的执行体验

- 使用 localStorage 保存当前 stop 勾选状态。
- 页面访问和地图点击统计。
- 移动端体验优化。
- 示例 route JSON。
- 更稳定的 schema 校验。

### V2：更多输入类型

- 在合规和可访问前提下解析公开网页。
- 支持截图或 PDF 行程单 OCR。
- 支持 Notion / 飞书 Markdown 导入。
- 更好的地点去重。
- 基于区域的轻量排序。

### V3：独立 Web 产品

- 网页表单。
- 托管 AI 后端或 serverless 生成服务。
- 用户账号。
- 路线模板库。
- 创作者分享页。

## 16. 面试叙事

项目可以这样讲：

> 我最初做的是一个静态 Travel Roadbook，用来服务自己的旅行。但进一步思考后，我发现更大的产品机会不是某一份路书页面，而是“把零散攻略、酒店和交通信息转成可执行路书”的生成流程。所以 V0 先聚焦稳定闭环：用户粘贴资料，Agent 提取结构化行程，生成 JSON，再渲染成网页路书。我刻意没有在第一版做账号导入、爬虫和实时路线优化，因为这些会增加不稳定性，而 MVP 最重要的是先证明生成链路和执行体验成立。

## 17. 实现边界

下一步实现计划只覆盖 V0：

- 创建示例输入格式。
- 定义 JSON schema。
- 构建渲染脚本。
- 复用现有 roadbook CSS。
- 生成一条 demo 路线。
- 添加简单校验。

在 V0 能从粘贴输入生成完整网页路书之前，不做 V1 或 V2 功能。
